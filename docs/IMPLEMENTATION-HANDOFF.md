The original plan ("Panel owns `goalDraft`/`ruleDrafts`/a mutation status", `key={talkId}` for "clean draft reset", TRAP-2 option **a**) all produced real remount regressions that `codex exec review` caught in sequence: stale-draft-revert, lost `'saving'` lockout → double-submit, lost unsaved drafts on tab switch, cross-talk draft clobber. **Resolution: keep state page-owned (like `jobDraft`) and make the panel presentational** — drafts + `contextStatus` live on the page; the panel gets them as props. TRAP 1 dissolved by the **sparse-override** model (render from the live prop, override only for actively-edited fields, cleared via id-keyed deletes); TRAP 2 resolved with option **b** (one page-owned status, threaded in — the original loading-first gate + tab-bottom success message preserved). Net: the panel ended ~15 props but behavior-identical. **`activeRuleCount` badge stays page-owned (reads `contextRules`); composer/`contextSources` untouched.**
3. ✅ **DONE (`a9f8e45`) — Slice 4: extract `TalkJobsPanel` (Jobs tab).** See the table row above. **Not yet Joseph-visual-verified** (behavioral UI refactor). **Local-stack click-list** (`db:start` + `dev:worker` :8788 + `dev:web` :5173, login via real Google OAuth): open a Talk → **Jobs** tab → (a) **New Job**, fill title/prompt/agent/schedule, **Save Job** → job appears in the list, "Job created."; (b) select it, edit the prompt, **switch to another tab and back** → your edit is still there (page-owned draft survives remount); (c) toggle a weekday + a tool-scope checkbox, **Save** → persists; (d) **Run Now** → status cycles queued→…→completed and Recent runs updates; (e) **Pause**/**Resume** flips status; (f) **Delete** → removed, next job auto-selected (or "Select or create a job" when none). Watch for: a job created/deleted while you tab away then back must reflect correctly (the in-flight-mutation race codex flagged), and the brief "Loading job tools…" flash on tab re-entry is expected. **Lesson that sharpens the Slice 2 rule:** "self-fetch the data like `TalkToolsPanel`" is only safe for **pure read-only data that no page-surviving handler ever writes** (drive bindings, tool scope). Any state an async **mutation** writes — a list mutated by save/delete, runs updated by a poll — must be **page-owned**, because the mutation can resolve *after* the tab-switch unmount and would otherwise land on an orphaned panel-local copy while the page-owned selection/status advance (codex caught exactly this). When in doubt, default to fully presentational + page-owned (load-once via a page-owned `…Loaded` flag so nothing re-fetches on remount).
